# V4L2 框架

## 1 概述

V4L2（Video for Linux 2）是 Linux 内核的视频设备标准框架，为应用程序提供统一的视频采集、输出和编解码接口。在 ESP32-P4 平台上，ESP-IDF 的 `esp_video` 组件在 V4L2 模型之上封装了 MIPI CSI 摄像头、ISP 和 video encoder 的驱动，通过标准文件操作（`open/close/ioctl/mmap`）暴露设备节点。

本项目使用 V4L2 的完整链路：
```
OV5647 sensor → MIPI CSI-2 → ESP32P4 ISP → V4L2 驱动 → Userspace (BSP/DAL)
```

核心设备节点：
| 设备节点 | 功能 | 本项目用途 |
|----------|------|-----------|
| `/dev/video0` | 视频采集（video capture） | 取帧 `DQBUF`、还帧 `QBUF` |
| `/dev/video20` | ISP 运行时控制 | BLC/BF/CCM/Gamma/WB 调参 |

## 2 编程模型

### 2.1 标准生命周期

```
open → G_FMT → S_FMT → REQBUFS → QUERYBUF → (mmap/malloc) → QBUF
                                                                │
                    ┌───────────────────────────────────────────┘
                    ▼
              STREAMON ──────────────────────────────┐
                    │                                │
                    ▼                                │
     ┌── DQBUF (取帧) ──→ 用户处理 ──→ QBUF (归还) ─┘
     │
     ... 循环采集 ...
     │
     ▼
  STREAMOFF → munmap/free → close
```

### 2.2 各步骤详解

#### 2.2.1 open — 打开设备

```c
// 非阻塞模式：DQBUF 无帧时返回 EAGAIN，不阻塞线程
int fd = open("/dev/video0", O_RDONLY | O_NONBLOCK);

// 阻塞模式：DQBUF 无帧时挂起等待
int fd = open("/dev/video0", O_RDWR);
```

本项目使用 `O_NONBLOCK`，由上层在自有任务上下文中轮询 DQBUF + 超时控制，详见 §3.1。

#### 2.2.2 G_FMT / S_FMT — 格式协商

V4L2 格式协商是双向的：应用请求一种格式，驱动可能根据硬件能力**调整**到最接近的支持值。因此 **S_FMT 后必须验证**返回的 `pixelformat` 是否与请求一致。

```c
// 1. 先读当前格式
struct v4l2_format vfmt = { .type = V4L2_BUF_TYPE_VIDEO_CAPTURE };
ioctl(fd, VIDIOC_G_FMT, &vfmt);

// 2. 覆盖 pixelformat
vfmt.fmt.pix.pixelformat = V4L2_PIX_FMT_RGB24;  // RGB888

// 3. 设置格式
ioctl(fd, VIDIOC_S_FMT, &vfmt);

// 4. 校验返回的 pixelformat（关键！）
if (vfmt.fmt.pix.pixelformat != V4L2_PIX_FMT_RGB24) {
    // 驱动不支持该格式，必须失败
    return error;
}
// 此时 vfmt.fmt.pix.width/height 是驱动实际配置的分辨率
```

常用 FourCC 编码（`pixelformat` 字段）：

| 宏 | 值 | 说明 |
|----|-----|------|
| `V4L2_PIX_FMT_RGB565` | `0x50424752` | RGB565（2 字节/像素） |
| `V4L2_PIX_FMT_RGB24` | `0x33424752` | RGB888（3 字节/像素） |
| `V4L2_PIX_FMT_JPEG` | `0x4745504A` | JPEG 压缩流 |
| `V4L2_PIX_FMT_SRGGB8` | `0x42474752` | Bayer RAW8 |

#### 2.2.3 REQBUFS — 申请帧缓冲

向驱动申请 N 个 DMA buffer，驱动将在内核侧分配内存（MMAP 模式）或接受用户空间指针（USERPTR 模式）。

```c
struct v4l2_requestbuffers req = {
    .count  = 3,                             // 缓冲数量（≥2 做流水线）
    .type   = V4L2_BUF_TYPE_VIDEO_CAPTURE,
    .memory = V4L2_MEMORY_MMAP,              // 或 V4L2_MEMORY_USERPTR
};
ioctl(fd, VIDIOC_REQBUFS, &req);
// req.count 返回驱动实际分配的缓冲数
```

#### 2.2.4 QUERYBUF — 查询单个缓冲信息

获取每个缓冲的物理偏移（MMAP 用）或长度信息。

```c
struct v4l2_buffer buf = {
    .type   = V4L2_BUF_TYPE_VIDEO_CAPTURE,
    .memory = V4L2_MEMORY_MMAP,
    .index  = i,
};
ioctl(fd, VIDIOC_QUERYBUF, &buf);
// buf.length   — 缓冲大小（字节）
// buf.m.offset — MMAP 偏移（仅 V4L2_MEMORY_MMAP）
```

#### 2.2.5 mmap / malloc — 映射用户空间

根据内存类型将缓冲映射到用户空间指针：

```c
// MMAP 模式：mmap 内核缓冲到用户空间
void *fb = mmap(NULL, buf.length, PROT_READ | PROT_WRITE,
                MAP_SHARED, fd, buf.m.offset);

// USERPTR 模式：用户自己分配
void *fb = heap_caps_malloc(buf.length, MALLOC_CAP_SPIRAM | MALLOC_CAP_DMA);
buf.m.userptr = (unsigned long)fb;
```

#### 2.2.6 QBUF — 缓冲入队

将空的缓冲提交到驱动的输入队列，供 DMA 引擎写入数据。

```c
ioctl(fd, VIDIOC_QBUF, &buf);
```

**init 阶段**：所有缓冲在 STREAMON 之前**全部 QBUF**，驱动才能开始填充。

**采集阶段**：处理完一帧后 QBUF 归还，驱动将此缓冲重新加入空闲队列。

#### 2.2.7 STREAMON / STREAMOFF — 流控制

```c
enum v4l2_buf_type type = V4L2_BUF_TYPE_VIDEO_CAPTURE;
ioctl(fd, VIDIOC_STREAMON, &type);   // 启动 DMA 采集
// ... 采集循环 ...
ioctl(fd, VIDIOC_STREAMOFF, &type);  // 停止 DMA 采集
```

本项目中 `STREAMON` 在 `init` 时执行一次后**永久保持**，不在每次 capture 时重置流。

#### 2.2.8 DQBUF — 取帧（核心操作）

从驱动的输出队列取出一个已被 DMA 填充完毕的缓冲。

```c
struct v4l2_buffer buf = {
    .type   = V4L2_BUF_TYPE_VIDEO_CAPTURE,
    .memory = V4L2_MEMORY_MMAP,
};
ioctl(fd, VIDIOC_DQBUF, &buf);

// 关键字段：
// buf.index      — 缓冲索引（用于定位 fb_bufs[index]）
// buf.bytesused  — 实际有效数据字节数
// buf.flags      — V4L2_BUF_FLAG_ERROR 等状态标志
// buf.timestamp  — 内核时间戳
```

**O_NONBLOCK 模式特殊处理**：
- 无就绪帧 → `errno == EAGAIN` → 短暂延时后重试
- 硬件错误 → `errno != EAGAIN` → 返回 `DAL_ERR_HW`

## 3 本项目中的 V4L2 实现

### 3.1 O_NONBLOCK 轮询 + 超时机制

文件：[bsp_camera_ov5647.c](software_engineering/firmware/components/bsp/bsp_camera_ov5647/bsp_camera_ov5647.c) — `ov5647_capture_frame()`（第 313-371 行）

```
capture_frame(ctx, timeout_ms):
  │
  ├── xSemaphoreTake(mutex)                    // 持锁保护
  │
  └── for (;;) {
        ioctl(DQBUF)
          ├─ 成功 ──→ 回写 v4l2_bufs[index]    // 供 return_frame 用
          │           frame_handle = index + 1   // 编码到 void *
          │           xSemaphoreGive(mutex)
          │           return DAL_OK
          │
          ├─ EAGAIN ──→ 检查超时
          │              timeout < 0 → 永久等（不检查）
          │              elapsed >= timeout → DAL_ERR_TIMEOUT
          │              未超时 → 释放 mutex → vTaskDelay(1ms)
          │                       → 重新持锁 → continue
          │
          └─ 其他错误 ──→ DAL_ERR_HW
      }
```

设计要点：
- **持锁期间 DQBUF**：防止与 `return_frame` 的 QBUF 并发竞争 `v4l2_bufs` 数组
- **EAGAIN 期间释放 mutex**：允许 `return_frame` 在等待期间并发归还帧，避免队列干涸死锁
- **1ms 轮询间隔**：平衡响应速度与 CPU 占用

### 3.2 帧归还的 frame_handle 编码

`frame_handle` 是 `void *` 类型，用于 return_frame 时定位要归还的缓冲。编码方式：

```c
frame_handle = (void *)(uintptr_t)(buf.index + 1U);
```

解码方式：

```c
uint8_t index = (uint8_t)(uintptr_t)frame_handle - 1U;
```

为什么不是 `index + 0`？因为 `index = 0` 时 `frame_handle = NULL`，而 `return_frame` 以 `handle == NULL` 为非法参数拒绝归还，导致槽位 0 的帧永远无法归还、V4L2 队列耗尽只剩 1 帧在循环。

### 3.3 双内存模式

| 模式 | 配置方式 | 缓冲来源 | 何时用 |
|------|----------|----------|--------|
| **MMAP** | 默认（`external_bufs=false`） | 内核驱动分配，`mmap()` 映射 | 通用场景，简单可靠 |
| **USERPTR** | `set_userptr_bufs(bufs, n, size)` 在 init 前调用 | 外部 PSRAM 缓冲（如 buffer_pool） | 零拷贝多消费者 pipeline |

USERPTR 模式下，`bsp_camera_ov5647_set_userptr_bufs()` 接受外部注入的缓冲指针数组：

```c
void bsp_camera_ov5647_set_userptr_bufs(void **bufs, uint8_t count, size_t buf_size);
```

`init()` 内检测 `external_bufs == true` 后自动走 USERPTR 路径，`heap_caps_malloc` 被跳过，直接用外部指针填入 `buf.m.userptr`。

### 3.4 ISP 运行时控制

文件：[bsp_camera_ov5647.c](software_engineering/firmware/components/bsp/bsp_camera_ov5647/bsp_camera_ov5647.c) — ISP 调参接口（第 667-819 行）

ISP 通过独立的 `/dev/video20` 设备节点控制，每次操作走 `open → ioctl(S_EXT_CTRLS) → close` 三个步骤：

```c
static dal_err_t isp_set_ctrl(int id, const void *data, size_t size)
{
    int fd = open("/dev/video20", O_RDWR);     // 每次 open/close
    struct v4l2_ext_control ctrl = {
        .id   = id,                             // V4L2_CID_USER_ESP_ISP_*
        .p_u8 = (uint8_t *)data,                // 结构体指针
        .size = size,                           // sizeof(结构体)
    };
    struct v4l2_ext_controls controls = {
        .ctrl_class = V4L2_CTRL_CLASS_USER,
        .count      = 1,
        .controls   = &ctrl,
    };
    ioctl(fd, VIDIOC_S_EXT_CTRLS, &controls);
    close(fd);
}
```

可控制的 ISP 模块：

| 模块 | 控件 ID | 结构体 |
|------|---------|--------|
| BLC（黑电平校正） | `V4L2_CID_USER_ESP_ISP_BLC` | `esp_video_isp_blc_t` |
| BF（双边滤波降噪） | `V4L2_CID_USER_ESP_ISP_BF` | `esp_video_isp_bf_t` |
| CCM（色彩校正矩阵） | `V4L2_CID_USER_ESP_ISP_CCM` | `esp_video_isp_ccm_t` |
| Gamma | `V4L2_CID_USER_ESP_ISP_GAMMA` | `esp_video_isp_gamma_t` |
| WB（白平衡） | `V4L2_CID_USER_ESP_ISP_WB` | `esp_video_isp_wb_t` |

**自动控制器行为**：ISP 自动控制器（3A：Auto Exposure / Auto White Balance / Auto Gain）持续运行。手动设置的 BLC 值稳定保持（不被控制器覆盖），而 BF/CCM/Gamma/WB 的设置会被控制器按增益/亮度自适应调整，手动覆盖为一次性生效。

### 3.5 RAW Bayer 帧采集 + BLC 标定

文件：[bsp_camera_ov5647.c](software_engineering/firmware/components/bsp/bsp_camera_ov5647/bsp_camera_ov5647.c) — `ov5647_capture_raw_frame()`（第 831-1083 行）

独立流程，与当前 RGB 流互斥：

```
1. G_FMT 保存当前格式/分辨率
2. STREAMOFF + 释放 MMAP 缓冲
3. S_FMT 切换为 V4L2_PIX_FMT_SRGGB8（RAW Bayer）
4. REQBUFS(1) + mmap + QBUF（仅 1 个 RAW 缓冲）
5. STREAMON + DQBUF（超时重试 200 次 × 10ms）
6. memcpy RAW 数据到用户缓冲区
7. STREAMOFF + munmap + REQBUFS(0)（释放 RAW 缓冲）
8. S_FMT 恢复原始格式
9. REQBUFS + QUERYBUF + mmap + QBUF（恢复原始帧缓冲）
10. STREAMON 恢复采集流
```

全程 ~200-500ms 无帧，BLC 标定算法分离 RGGB 四通道计算均值：

```
Bayer RGGB 阵列：         通道分离逻辑：
  R  Gr  R  Gr           (y&1)==0, (x&1)==0 → R
  Gb B  Gb B             (y&1)==0, (x&1)==1 → Gr
  R  Gr  R  Gr           (y&1)==1, (x&1)==0 → Gb
  Gb B  Gb B             (y&1)==1, (x&1)==1 → B
```

## 4 内存模型对比

### 4.1 MMAP（Memory Mapping）

```
┌──────────┐     ┌──────────────┐     ┌───────────┐
│ Userspace │◀───│ mmap 映射    │◀───│ Kernel    │
│  fb_bufs  │    │ PROT_READ|WR │    │ DMA 缓冲   │
│  指针数组  │    │ MAP_SHARED   │    │ (物理连续) │
└──────────┘     └──────────────┘     └───────────┘
```

- **驱动分配、用户映射**：内存由内核 DMA 分配器管理
- **零拷贝读**：用户直接读 DMA 缓冲，无需 CPU memcpy
- **缺点**：缓冲归驱动所有，不能跨消费者共享

### 4.2 USERPTR（User Pointer）

```
┌──────────────┐     ┌──────────┐     ┌───────────┐
│ buffer_pool  │────▶│ Userspace│────▶│ Kernel    │
│ (PSRAM, DMA) │     │ 指针注入  │     │ DMA 写入  │
└──────────────┘     └──────────┘     └───────────┘
```

- **用户分配、驱动写入**：DMA 直接写入用户提供的 PSRAM 缓冲
- **真正零拷贝**：采集帧和消费者看到的是**同一块物理内存**
- **适用场景**：多消费者 pipeline（UI 预览 + 人脸检测同时持有一帧的数据指针）

### 4.3 DMABUF（跨设备共享，本项目未使用）

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  camera  │────▶│ dmabuf fd│◀────│ display  │
│ /dev/vid0│     │ (共享句柄)│     │ /dev/dri │
└──────────┘     └──────────┘     └──────────┘
```

用于 GPU/display 需要**直接消费**摄像头数据的场景（如 Android HAL3）。

## 5 V4L2 Buffer 状态机

```
    ┌───────┐
    │ FREE  │ ◀──────────────────────────────────┐
    │ (空闲) │                                     │
    └───┬───┘                                     │
        │ REQBUFS 分配                            │
        ▼                                         │
    ┌──────────┐    QBUF (入队)    ┌───────────┐  │
    │ DEQUEUED │ ─────────────────▶│  QUEUED   │  │
    │ (用户持有)│                   │ (驱动持有) │  │
    └──────────┘ ◀─────────────────└───────────┘  │
         ▲           DQBUF (出队)        │         │
         │                               │ DMA 填充 │
         │                               ▼         │
         │                          ┌───────────┐  │
         └──────────────────────────│   DONE    │──┘
                  QBUF (归还)       │ (数据就绪) │
                                    └───────────┘
```

关键规则：
- **REQBUFS 后的初始状态是 DEQUEUED**，必须 `QBUF` 才能被 DMA 使用
- **DQBUF 只能对 DONE 状态的缓冲操作**
- **QBUF 后缓冲从 DEQUEUED 变为 QUEUED**，驱动接管
- **驱动内部** QUEUED → DMA 填充 → DONE → 等待 DQBUF 取出

## 6 并发模型与同步

### 6.1 当前实现：单 Mutex + 释放-重持锁

```c
capture_frame():                    return_frame():
  take(mutex)                         take(mutex)
  for (;;) {                          QBUF (归还帧)
    DQBUF (持锁)                      give(mutex)
    if (EAGAIN) {
      give(mutex)  ←── 关键：释放锁让路
      vTaskDelay(1ms)
      take(mutex)  ←── 重新持锁继续
      continue;
    }
    填 frame 字段
    give(mutex)
    return DAL_OK
  }
```

**为什么不用 Semaphore？**

Semaphore 是计数信号量，允许多个释放。这里需要的是互斥访问 `v4l2_bufs` 数组——QBUF 和 DQBUF 不能同时操作同一个 `v4l2_buffer` 结构体，Mutex 是正确选择。

### 6.2 队列深度与帧丢弃

```c
// V4L2 缓冲数
#define BOARD_CAMERA_BUFFER_COUNT  3   // 驱动侧 DMA 队列

// 消费者队列深度（msg_queues.c）
g_q_cam_to_ui  = xQueueCreate(3, sizeof(frame_buffer_t))  // UI 端
g_q_cam_to_det = xQueueCreate(2, sizeof(frame_buffer_t))  // 检测端
```

**队列满时的新帧**：`xQueueSend(..., 0)` 非阻塞发送，队列满直接丢弃，帧被 `return_frame` 归还驱动。不会阻塞采集管线。

## 7 错误处理

| errno | 含义 | 本项目处理 |
|-------|------|-----------|
| `EAGAIN` / `EWOULDBLOCK` | DQBUF 无就绪帧（O_NONBLOCK） | 延时 1ms 重试，超时返 `DAL_ERR_TIMEOUT` |
| `EINVAL` | 参数非法 | `DAL_ERR_INVALID` |
| `ENOMEM` | 内存不足 | `DAL_ERR_NO_MEM` |
| `ENODEV` | 设备不存在 | `DAL_ERR_HW` |
| `EBUSY` | 设备忙（如重复 STREAMON） | `DAL_ERR_BUSY` |
| `EBADF` | 文件描述符无效 | `DAL_ERR_STATE` |

ESP-IDF 的 `esp_video` 框架错误码通过 `dal_err_from_esp()` 翻译，原始 `esp_err_t` 不透传到 DAL 以上层级。

## 8 性能特性

| 参数 | 值 | 说明 |
|------|-----|------|
| DQBUF 系统调用开销 | ~5-10µs | O_NONBLOCK 轮询模式下每次重试都要进内核 |
| 帧间隔 | ~33ms（30fps） | OV5647 最大帧率，RGB888 @ 800×480 |
| 单帧数据量 | 800×480×3 = 1.15MB | RGB888 格式 |
| PSRAM 带宽 | ~800MB/s（ESP32P4） | 足以支撑 30fps × 1.15MB ≪ 带宽 |
| mutex 持锁时间 | <100µs（非等待路径）| 只保护 struct 字段读写 |
| RAW 帧采集耗时 | 200-500ms | 切换格式 + 重建流，仅在标定时使用 |

### 优化方向

1. **阻塞 DQBUF 代替轮询**：`open(O_RDWR)` + `select/poll` 等帧就绪，省去 EAGAIN 循环的 CPU 浪费
2. **DMA 完成中断通知**：注册 ISP DMA 完成回调 → `xTaskNotifyFromISR`，绕开 DQBUF 系统调用
3. **增大 V4L2 缓冲数**：当前 3 个，可增至 4-6 个，提升流水线深度，应对消费者偶发慢速
